🛡️ Wireless Hardening Best Practices

▼
  • Use WPA3 if available; WPA2 is acceptable
  • Use a complex password / passphrase
  • Disable WPS (Wi-Fi Protected Setup) — it is insecure
  • Change default router credentials
  • Keep firmware updated
  • Consider hiding or changing your SSID
  • Enable a firewall on the router
  • Use a guest network for visitors (network segmentation)
WPS = insecure, disable it. Default credentials = risk. Guest network = segmentation for security.

📈 CVSS — 3 Scoring Metrics ANSWER CHOICES ON TEST

▼
Your instructor flagged these as exam answer choices — most people don't know them. The test will list all three and ask you to identify one.
MetricWhat it measuresKey questionChanges?
BaseIntrinsic characteristics of the vulnerability itself"How bad is this vulnerability in general?"Never — always present, most important
TemporalFactors that change over time (e.g. public exploit released, patch available)"What is the current threat level right now?"Yes — changes as exploits/patches emerge

Base = always present, most important. It is the foundation every CVSS score is built on.

📄 Why CVE Exists — The Common Name Problem

▼

Without CVE, different security vendors, tools, and professionals would name the same vulnerability differently — creating confusion and communication failures.

CVE solves this: Every publicly known vulnerability gets one unique identifier (CVE-YYYY-NNNNN) so all tools, vendors, and professionals worldwide refer to the exact same issue using the same name — no confusion, no duplication.
Without CVEWith CVE
Vendor A calls it "Apache RCE Bug #4"Everyone calls it CVE-2021-41773
Vendor B calls it "Log4j Remote Exploit"Everyone calls it CVE-2021-44228
Confusion, missed patches, tool mismatchesUniversal standard — patch management and scanning align globally

🌐 Internet-Facing Triage — The CVSS Trap TRICK QUESTION

▼
The trap: A question shows two servers — one with CVSS 9.8 and one with CVSS 8.x. You instantly want to pick 9.8.

The trick: Check the IP address. If the 9.8 server has a private IP (internal, not internet-facing), it is less accessible to outside attackers.

The 8.x server with a public/internet-facing IP gets priority — it is exposed to the entire internet and therefore carries greater immediate risk.
FactorWhy it matters for triage
CVSS ScoreSeverity of the vulnerability itself — higher = more dangerous in isolation
Internet-facing (public IP)Exposed to external attackers — massively increases real-world exploitability
Internal (private IP)Attacker must already be inside your network — lower immediate exposure
Asset valueHigh-value system (DB server, domain controller) elevates priority even with lower CVSS
Private IP ranges (internal only): 10.x.x.x  |  172.16–31.x.x  |  192.168.x.x  — anything else is public/internet-facing.
Exam rule: CVSS score is your starting point, but internet-facing + lower CVSS beats internal + higher CVSS when determining what to remediate first.

✅ True/False Positives & Negatives ON TEST

▼
Master rule: TRUE = scanner working correctly.   FALSE = scanner NOT working correctly.
ResultRealityScanner saysScanner statusDanger level
True PositiveReal threat EXISTSThreat detected ✓Working correctlyN/A — caught it
True NegativeNo threatNothing reported ✓Working correctlyN/A — all clear
False PositiveNo real threatReports a threat ✗Scanner failedWastes time chasing ghosts
False NegativeReal threat EXISTSReports nothing ✗Scanner failedMOST DANGEROUS
False Negative = worst outcome. A real threat is present but the scanner misses it entirely — you have no idea you're compromised.

Why False Negatives Happen

  • Scanner not updated — most common cause; signatures are outdated
  • Zero-days — new/unknown vulnerabilities not yet in the tool's database
  • Incomplete scans — unauthenticated scan misses vulnerabilities that require credentials to detect
Solution for incomplete scans → use Credentialed Scans. The scanner logs in with valid credentials and can see deeper into the system, dramatically reducing false negatives.

Why False Positives Happen

  • Generic/overly broad detection rules
  • Misconfigured scanner baselines
  • Outdated baselines that don't account for new custom applications

📄 STIX — Structured Threat Information eXpression TEST

▼

STIX is a standardized language / format for describing cyber threat intelligence (CTI). Think of it as the vocabulary — a common way for organizations to write down and share what they know about threats.

STIX = the FORMAT / LANGUAGE. It defines WHAT you're saying about a threat.
STIX DescribesExamples
Threat ActorsWho is attacking (nation-state, criminal group, insider)
Attack PatternsTTPs — Tactics, Techniques, and Procedures (linked to MITRE ATT&CK)
MalwareMalware families, behavior, capabilities
Indicators (IOCs)IP addresses, file hashes, domain names that indicate compromise
VulnerabilitiesCVEs and weaknesses being exploited
Courses of ActionRecommended mitigations or responses
CampaignsA group of related attacker activity over time

STIX Technical Facts

  • Format: JSON-based
  • Maintained by: OASIS (open standards organization)
  • Current version: STIX 2.x
  • Two object types: SDOs (STIX Domain Objects — the "things") and SROs (STIX Relationship Objects — how things connect)

🕒 TAXII — Trusted Automated eXchange of Intelligence Information TEST

▼

TAXII is the transport protocol / delivery mechanism for sharing STIX data. Think of it as the postal system — STIX is the letter, TAXII is how it gets delivered.

TAXII = the TRANSPORT / HOW you share it. STIX data travels over TAXII.
FeatureDetail
TransportUses HTTPS — secure, encrypted delivery
Collection modelRequest/response — clients query a TAXII server for available threat data
Channel modelPublish/subscribe — producers push updates to subscribers automatically
What it carriesPrimarily STIX objects (but technically transport-agnostic for other CTI formats)
Who uses itISACs (Information Sharing and Analysis Centers), government agencies, security vendors, enterprises

🔎 Penetration Testing, Bug Bounties & Audits ON TEST

🔫 Penetration Testing TEST

▼

A penetration test actively attempts to exploit vulnerabilities found by a vulnerability scanner to confirm whether they are truly exploitable in your environment.

PropertyDetail

Vuln scan ≠ pen test. Vuln scan = theoretical weaknesses. Pen test = confirmed exploitable vulnerabilities. The exam will test this distinction.

⚙️ Fine-Tuning a Scanner TRICK QUESTION ON TEST

▼

This shows up as a scenario question. A scanner (email scanner, vulnerability scanner, etc.) is missing things for a specific group of people — false negatives, filtering gaps, or certain content getting through.

The trap: Most answer choices say things like "tell each individual to do this" or "have each user adjust their settings."

The correct answer: Go into the scanner itself and fine-tune it — adjust the scanner's configuration/rules at the admin level.

Individual-level fixes are band-aids. The scalable, correct solution is to fix the scanner configuration centrally.
Wrong answers (distractors)Correct answer
"Tell each individual to configure their settings"Fine-tune / adjust the scanner configuration at the admin/system level
"Have each user whitelist the missed items"
"Ask each person to manually review what was missed"
Memory rule: If the question involves a scanner missing something for a GROUP of people — the answer is always to go into the scanner and fine-tune it, not to tell individuals to work around it.

💲 Quantitative Risk: AV · EF · SLE · ARO · ALE ON TEST

▼
SLE = AV × EF
ALE = SLE × ARO
Full chain:   ALE = AV × EF × ARO

The Five Terms

AcronymFull NameWhat It Means
AVAsset ValueTotal dollar value of the asset (system, data, equipment). Example: database worth $100,000
EFExposure FactorPercentage of the asset lost in a single incident — expressed as a decimal. 50% loss = 0.5
SLESingle Loss ExpectancyCost of one incident. Formula: SLE = AV × EF
AROAnnual Rate of OccurrenceHow many times the event is expected to happen per year. See trap below.
ALEAnnual Loss ExpectancyTotal yearly expected loss. Formula: ALE = SLE × ARO

Worked Example (from your notes)

StepCalculationResult
Asset Value (AV)Given$100,000
Exposure Factor (EF)Given: 50% loss per incident0.5
SLE = AV × EF$100,000 × 0.5$50,000 per incident
AROGiven: 2 times per year2
ALE = SLE × ARO$50,000 × 2$100,000 per year
THE ARO TRAP — This is where they get you on the exam:

ScenarioARO value
Happens once per year1
Happens twice per year2
Happens once every 2 years0.5
Happens once every 5 years0.2  ← exam trap!
Happens once every 10 years0.1
Rule: ARO = 1 ÷ number of years. "Once every 5 years" = 1/5 = 0.2, NOT 5.

Why This Matters for Security Decisions

Quantitative risk analysis uses AV and EF to get SLE, then multiplies by ARO to get ALE. ALE lets organizations estimate annual financial risk and decide whether the cost of a security control is justified — if the control costs less than the ALE, it's worth implementing.

Security control justification: If ALE = $100,000/year and a control costs $20,000/year — implement the control. If the control costs $150,000/year — the math says accept the risk.

When you see the word INSURANCE anywhere in the question — stop reading the rest, find Risk Transference in the answers. That is your answer.

Response choice depends on: risk severity · business impact · cost of fix · organizational risk threshold · available resources. Responses are managed through a formal change management process.

💰 Cyber Insurance = Risk Transference ON TEST

▼
PropertyDetail
What it doesShifts the financial impact of a security incident to a 3rd party (the insurer)
What it does NOT doDoes NOT reduce the likelihood of an attack — only reduces the financial burden if one occurs
CoversData breach costs, ransomware recovery, legal fees, regulatory fines, recovery efforts
QualificationOrganizations must demonstrate proper security controls and risk management practices to qualify — just having insurance doesn't mean you skip controls
Insurance ≠ security. You still must implement controls to qualify for coverage. Think of it like home insurance — the inspector checks your house before they'll cover it.

🔌 Patching — Proactive Remediation

▼

Patching = applying updates, fixes, or improvements to software, OS, or firmware to address known vulnerabilities. One of the most common forms of remediation — directly eliminates the underlying flaw.

Patch Management Process

StepAction
1. IdentifyFind missing patches via vulnerability scanning
2. TestTest patches in a controlled environment — avoid breaking production systems
3. DeployApply through the change management process
4. VerifyConfirm the vulnerability has been resolved
Poor or delayed patching is a leading cause of security breaches. Effective patch management reduces attack surface and prevents malware infections and unauthorized access.

🌐 Network Segmentation — Mitigation Strategy

▼

Segmentation divides a network into smaller isolated sections (subnets/zones) to limit the spread of attacks and control access between systems.

PropertyDetail
PurposeIf one segment is compromised, the attacker cannot easily move laterally to other areas
Tools usedFirewalls, VLANs, access controls, network policies
Response typeMitigation — reduces exposure and impact, does NOT eliminate the vulnerability itself
ProtectsHigh-value assets and sensitive data from general user networks

👥 Compensating Controls ON TEST

▼

Compensating controls are alternative security measures put in place when the primary control cannot be implemented — due to cost, technical limitations, or operational constraints.

PropertyDetail
PurposeReduce risk when you can't fully eliminate the vulnerability — provides a different layer of protection that achieves a similar objective
When usedLegacy systems · uptime requirements · business constraints that prevent immediate patching
ExamplesCan't patch immediately → add firewall rules + network segmentation + increased monitoring until the patch can be applied
Role in complianceEspecially important for maintaining compliance and security posture within acceptable risk thresholds
Compensating controls = temporary mitigations that buy time until the real fix (remediation/patching) can be applied — common in environments with legacy systems or strict uptime requirements.

🔆 Impact Types — 7 Categories ON TEST

▼

Impact = the damage or consequences if a vulnerability is exploited. Focuses on how bad the outcome is, not how likely it is to happen.

#Impact TypeWhat it coversExamples
1FinancialDirect monetary lossTheft, fraud, recovery costs, lost revenue

4Legal / RegulatoryCompliance consequencesFines, penalties, lawsuits, regulatory sanctions

Impact ≠ Likelihood. A vulnerability with HIGH impact but LOW likelihood still demands attention — impact drives how bad the outcome is if it happens. Both factors feed into Risk = Likelihood × Impact.
