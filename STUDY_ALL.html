Asymmetric Rules — Memorize These!

  • Only the private key can decrypt data encrypted with the public key
  • Only the public key can decrypt data encrypted with the private key
  • The public key verifies but does NOT decrypt a private key's signature

📨 Attack Vectors

▼

Message-Based Attacks

  • Email (Phishing) — most common; fake emails luring victims into providing info or clicking links
  • SMS / Smishing — phishing via text message. If you see SMS on the test = Smishing!
  • Vishing — voice phishing; talking someone into doing something over a phone call

File-Based Attacks

  • Embedded macros, malicious PDFs, Office macros, Trojan installers
  • Fileless malware — no file stays on the system; a downloader pulls malware, runs it, then removes itself
  • Images can be malicious via steganography — hidden malware or exfiltrated data inside image files

Removable Media

  • USB drives, SD cards, external HDDs can exfiltrate data or introduce malware
  • Even air-gapped systems can be compromised via removable media (e.g., Stuxnet on Iranian nuclear facility)
For exam: Disable AutoRun, restrict USB devices, user training

▼

  • Impossible Travel — login from New York then Paris 10 minutes later

🛡️ Hardening, Patching & Endpoint Protection

▼

Hardening

  • Securing a system by reducing its attack surface — removing unnecessary things from default state
  • Disable unnecessary ports and protocols
  • Change default passwords
  • Remove unnecessary software

Patching

  • Systematic notification, identification, deployment, installation, and verification of updates
  • Updates = patches, hotfixes, service packs
  • Must be kept current to address newly discovered vulnerabilities

  • Passive sensor — monitors traffic without interfering (like NIDS; no impact on performance)
  • Inline sensor — sits directly in the traffic path; can support prevention actions (like NIPS)

🌐 Internet-Facing Triage — The CVSS Trap TRICK QUESTION

▼
The trap: A question shows two servers — one with CVSS 9.8 and one with CVSS 8.x. You instantly want to pick 9.8.

The trick: Check the IP address. If the 9.8 server has a private IP (internal, not internet-facing), it is less accessible to outside attackers.

The 8.x server with a public/internet-facing IP gets priority — it is exposed to the entire internet and therefore carries greater immediate risk.
FactorWhy it matters for triage

🔥 Firewalls — Types, Rules & ACLs ON TEST

Firewall Rules — Critical Exam Details

RuleWhat it means
Top-to-bottom readingFirewalls process rules in order from top to bottom — the first matching rule wins
Implicit DenialThe last rule should always be deny all — any IP, any source, any destination, any port, any protocol not explicitly permitted is denied. This is implicit denial.

🔐 Access Control Models — MAC, DAC, RBAC, ABAC, Rule-Based ON TEST

▼

Access control models define how permissions are assigned and enforced so that users and systems only access what they are authorized to use. Each model approaches this differently based on control, flexibility, and security requirements.

Quick reference — memorize these distinctions:

MAC = centrally controlled + classification labels  |  DAC = owner controls  |  RBAC = roles  |  ABAC = attributes  |  Rule-Based = system rules
ModelFull NameWho controls accessKey characteristicTypical use
MAC Mandatory Access Control Centrally controlled — the system/admin, not the owner Based on classification levels or labels (e.g., Unclassified, Secret, Top Secret). Users cannot change permissions — the system enforces them. Government, military, high-security environments
DAC Discretionary Access Control Resource owner controls who gets access Owner has discretion to grant or revoke access to their own resources. Most flexible — but most risk if owner makes a mistake. Standard file systems (Windows NTFS, Linux file permissions)
RBAC Role-Based Access Control Access assigned based on job roles Users inherit permissions from their role (e.g., "Nurse" role gets access to patient records; "HR" role gets access to HR systems). Easy to manage at scale. Enterprise environments, Active Directory groups
ABAC Attribute-Based Access Control Access based on dynamic attributes Uses multiple attributes — user attributes (department, clearance), resource attributes (sensitivity), and environment attributes (time of day, location) — to make flexible, context-aware decisions. Cloud environments, zero trust architectures
Rule-Based Rule-Based Access Control Access enforced by system-defined rules Static rules set by administrators that apply to all users uniformly — e.g., "No access to the network after 6 PM" or firewall ACL rules. Rules are not tied to identity or role. Firewalls, routers, time-based access restrictions

MAC vs DAC — The Key Difference

MAC (Mandatory):
Owner cannot change permissions. The system enforces labels. Even if you own the file, you cannot grant access to someone with a lower classification. Control is mandatory — taken away from the user.
DAC (Discretionary):
Owner decides who gets access. You own the file, you choose who can read it. Control is discretionary — left to the owner's judgment.
RBAC vs ABAC: RBAC grants access based on a static role assigned to the user. ABAC is more dynamic — it evaluates multiple attributes in real time (who you are + what you're accessing + when/where you're doing it) to make the access decision. ABAC is more flexible and granular but more complex to configure.

🔒 Privileged Access & PAM ON TEST

▼

Privileged access refers to elevated permissions that allow a user or account to perform high-level actions — modifying configurations, installing software, accessing sensitive data, and managing other user accounts. Privileged users include system admins, network engineers, DBAs, and service accounts.

Privileged accounts = high-value attack targets. If compromised, an attacker gains extensive control over the entire environment. They must be tightly controlled and continuously monitored.

PAM — Privileged Access Management

PAM is the overarching framework that controls, monitors, and audits privileged accounts and activities. It integrates three key controls:

ControlWhat it doesWhy it matters
Temporal Accounts Accounts that exist only for a limited time — automatically disabled or removed after a defined period Ensures elevated access does not persist beyond what is necessary for the task. No lingering accounts = smaller attack surface.
Password Vaulting Privileged credentials stored in a secure encrypted vault — users never directly know or handle the passwords. They request access and the system grants it. System controls, rotates, and audits credential use. Prevents credential reuse and exposure. Admins can't share or leak passwords they never see.
JIT — Just-in-Time Permissions Elevated access is dynamically granted only when needed and only for a short duration, then automatically revoked Aligns with least privilege — accounts are unprivileged by default and elevated only at the moment of need. Significantly reduces the attack window.
Memory rule — PAM = Vault + Temp + JIT. Password vaulting (secrets secured), Temporal accounts (time-limited), Just-in-Time permissions (on-demand elevation). Together they enforce least privilege for the most dangerous accounts in the environment.

1. PreparationEstablish plans, tools, policies, and train personnel before an incident occurs
2. Detection & AnalysisIdentify and investigate the incident — determine scope and impact
3. ContainmentTake action to limit the spread or damage — isolate affected systems
4. EradicationRemove the root cause — malware, unauthorized access, misconfiguration — from the environment
5. RecoveryRestore systems to normal operations securely

IR integrates with SIEM (detection and logging) and SOAR (automated response) for faster, more consistent incident handling.

🔍 Root Cause Analysis (RCA) ON TEST

▼

RCA is the process of identifying the underlying reason why a security incident occurred — not just fixing symptoms, but finding what actually failed.

FeatureDetail
When it happensAfter the incident is contained and resolved — part of the Lessons Learned phase
What it examinesLogs, system behavior, and response actions to pinpoint the origin
Common root causesMisconfiguration, unpatched vulnerability, weak access controls, human error
GoalImplement long-term fixes, update policies, enhance training — prevent the same incident from happening again
RCA ≠ fixing symptoms. Rebooting a compromised server is fixing a symptom. Finding the unpatched vulnerability that allowed the compromise = root cause analysis.

💻 Digital Forensics — Core Concepts ON TEST

▼

Digital forensics is the process of collecting, analyzing, and preserving digital evidence from computers, networks, and devices in a manner that maintains integrity — so it can be used in investigations or legal proceedings.

Order of operations: Legal Hold → Chain of Custody → Acquisition → Preservation → Analysis → Reporting
ConceptWhat it means
Legal HoldWhen litigation or investigation is anticipated, normal data handling (deletion, overwriting, log rotation) must be SUSPENDED. Preserve all potentially relevant data — failure to do so can result in legal penalties.
Chain of CustodyDocuments every person who collected, handled, accessed, or transferred evidence — from collection to court. Proves evidence was not altered or tampered with. Critical for admissibility.
Forensic ImageExact bit-by-bit copy of the original data. Always work on the copy — never the original. Original goes into a Faraday cage (static-safe, electromagnetically shielded storage).
Write BlockerHardware or software tool that prevents any writes to original media during acquisition. Ensures no data is accidentally written, altered, or deleted. Critical for legal admissibility.
HashingGenerate MD5 or SHA-256 hash before and after acquisition. If hashes match, data is unchanged and trustworthy — proves integrity of the forensic image.
DCFLDDDoD Computer Forensic Lab Disk Duplication tool — has three modes: copy, hash, and forensic image. Always use forensic image — it copies bit-by-bit AND hashes simultaneously.
PreservationProtecting evidence in its original state throughout the process — forensic images instead of originals, write blockers, proper environmental and access controls for storage
ReportingFinal phase — documents what happened, how, what was affected, actions taken, and recommendations. No jargon, no opinions — factual and tailored to the audience (technical teams, management, or courts)

📜 Governance & Policy

Security Governance Overview

Security Governance = the framework of policies, procedures, roles, and responsibilities that govern how an organization manages and enforces its overall security strategy.

  • Establishes WHO is responsible, HOW decisions are made, WHAT standards must be followed
  • Ensures security aligns with business objectives, regulatory requirements, and risk management
  • Includes: creating policies, defining accountability, ensuring compliance through audits
  • Purpose: provide structured direction so security is treated as a business-level priority, not just a technical function
Policy Hierarchy — Policy vs Standard vs Guideline vs SOP
Type Mandatory? What It Covers Test Keyword
Policy YES High-level rule — WHAT must be done and WHY. Created by management. Broad and enforceable. WHAT / WHY
Standard YES Specific technical requirements supporting the policy (encryption algorithm, password length, config settings). Little room for interpretation. SPECIFIC / TECHNICAL
Guideline NO Recommended best practices — advisory, flexible, allows professional judgment. NOT mandatory. RECOMMENDED / FLEXIBLE
SOP (Standard Operating Procedure) YES Detailed step-by-step instructions for carrying out tasks in alignment with policies. HOW to execute. HOW / STEP-BY-STEP

⚠ TEST TIP: Policy = WHAT and WHY (mandatory rule). SOP = HOW (step-by-step execution). Together, policies set the rules and SOPs ensure those rules are followed consistently.

Agreements & Contracts — BPA, SLA, MOA, MOU, ISA
Agreement Full Name Focus Key Distinction
BPA Business Partnership Agreement Security responsibilities & obligations between org and 3rd party partner SECURITY and TRUST in the partnership
SLA Service Level Agreement Measurable performance metrics: uptime, availability, response times, support expectations, remedies if not met PERFORMANCE and SERVICE RELIABILITY — keyword: UPTIME
MOA Memorandum of Agreement Formal, specific responsibilities, roles, deliverables — resource sharing, funding, services MORE FORMAL, binding — similar to a contract
MOU Memorandum of Understanding Mutual understanding of shared goals/intentions — general roles without strict enforcement LESS FORMAL — no legal weight of a contract
ISA Interconnection Security Agreement Security requirements for CONNECTED IT systems between organizations: auth methods, encryption, access controls, incident response SPECIFICALLY for securing system-to-system connections and data exchange

⚠ TEST TIP: BPA = security & trust | SLA = performance & uptime | MOA = formal/enforceable | MOU = informal/mutual | ISA = securing connections

Acceptable Use Policy (AUP)

AUP = a formal, mandatory, and enforceable document defining the rules and guidelines for how users are allowed to use an organization's IT systems, networks, and resources.

  • Outlines permitted business use and prohibited activities (unauthorized access, downloading malicious content)
  • Expectations for protecting company data
  • Consequences for violations clearly stated
  • Reduces security risk by communicating user responsibilities
  • Ensures technology is used safely, responsibly, and in alignment with security policies
IR vs Business Continuity vs Disaster Recovery — BCDR
Concept Focus Scope
IR (Incident Response) Detect, contain, eradicate, recover from active security events REACTIVE — short-term, directly deals with active threats
Business Continuity (BC) Ensure critical business functions continue during ANY disruption (cyber, natural disaster, system failure) BROADER — planning, redundancy, risk management to maintain essential services
Disaster Recovery (DR) Restore IT systems, data, and infrastructure after major disruption — rebuild servers, recover from backups, re-establish network SUBSET of BC — guided by RTO (Recovery Time Objective) and RPO (Recovery Point Objective)

⚠ ON TEST: You may see BCDR (Business Continuity and Disaster Recovery combined) — it is the same concept, combined for ease of management.

KEY DISTINCTION: IR handles ACTIVE security events | BC ensures operations CONTINUE | DR RESTORES systems and data after disruption

Change Management & Change Control

Change Management = a structured process to plan, review, approve, implement, and document changes to systems, networks, or applications in a controlled and secure manner.

Goal: minimize risk of disruption, prevent unintended consequences, ensure all changes are properly tested and authorized before being applied.

Change Management Process Steps:

  1. Request a Change — submit a change request (CR or RFC)
  2. Risk & Impact Assessment — evaluate potential risks and effects
  3. Approval — Change Advisory Board (CAB) — formal approval body
  4. Test in Controlled Environment — validate before production
  5. Implement the Change — deploy to production
  6. Document the Results — for accountability and future reference

Change Control = the formal sub-process within change management focused on PREVENTING disruptions, security vulnerabilities, and unintended consequences caused by unmanaged changes.

SDLC — Software Development Life Cycle

SDLC = a structured process to design, develop, test, deploy, and maintain software in a secure and systematic way.

SSDLC (Secure SDLC) = security integrated throughout every phase — NOT as an afterthought. Fixing issues early reduces risk and cost.

SDLC Phases:

Phase What Happens
1. PlanningDefine goals, scope, requirements, resources, timelines, potential risks — sets foundation
2. DesignCreate system architecture, data flows, security controls — security built IN not added later
3. Development / ImplementationActual coding — follow secure coding practices to minimize vulnerabilities
4. TestingFunctional testing + security testing — ensure software works correctly and securely
5. DeploymentRelease software to production environment for end users
6. MaintenanceOngoing updates, patches, vulnerability fixes, performance improvements
Procedures — Onboarding, Offboarding & Playbooks

Procedures = detailed step-by-step instructions that explain exactly HOW to perform specific tasks in alignment with policies and standards.

Onboarding

Integrating NEW users into org systems:

  • Create accounts
  • Assign access based on least privilege
  • Provide security training
  • Ensure user understands AUP

Critical for establishing secure access from the start

Offboarding

Securely removing DEPARTING users:

  • Disable or delete accounts
  • Revoke ALL permissions
  • Recover assets
  • Ensure NO unauthorized access remains

Prevents insider threats and lingering access risk

Playbooks

Predefined structured procedures for SPECIFIC incidents (malware, phishing):

  • Provides clear, repeatable guidance
  • Integrated into SOAR automation
  • SOAR uses BOTH playbooks AND runbooks

Fast, consistent, automated incident response

Data Sources — Automated Reports, Dashboards & PCAP
  • Automated Reports — system-generated summaries from SIEM or vulnerability scanners highlighting findings, trends, and alerts. Helps analysts quickly prioritize responses.
  • Dashboards — real-time visual representation of security data aggregating from multiple sources into charts, graphs, and alerts. Enables quick decision-making and situational awareness.
  • PCAP (Packet Capture) — detailed recordings of network traffic allowing analysts to inspect individual packets. Especially valuable for deep analysis of attacks, troubleshooting, and forensic investigation.

Together these provide visibility into system health, network activity, and potential threats — allowing organizations to detect vulnerabilities, monitor ongoing activity, and respond effectively.

Legal Considerations

Legal Considerations = the laws, regulations, and legal obligations governing how an organization manages security, data, and incident response activities.

  • Compliance with applicable regulations: data protection laws, privacy requirements, industry standards
  • Respecting user privacy and properly handling PII (Personally Identifiable Information)
  • Ensuring monitoring or investigative actions are legally authorized
  • Critical during incident response and digital forensics — evidence collection must be legally sanctioned
