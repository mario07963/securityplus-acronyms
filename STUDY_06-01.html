⚖️ Symmetric vs Asymmetric Encryption

⚠ ON TEST ▼
SymmetricAsymmetric
KeysSame key to encrypt and decryptTwo keys — public + private

Asymmetric Rules — Memorize These!

  • Only the private key can decrypt data encrypted with the public key
  • Only the public key can decrypt data encrypted with the private key
  • The public key verifies but does NOT decrypt a private key's signature

Key Exchange

MethodDescription
Out-of-BandUses a separate, independent channel — more secure
In-BandSame channel as data — vulnerable to on-path / MITM attacks

Key Length = Key Strength. Asymmetric keys should be longer than symmetric keys.

📨 Attack Vectors

▼

Message-Based Attacks

  • Email (Phishing) — most common; fake emails luring victims into providing info or clicking links
  • SMS / Smishing — phishing via text message. If you see SMS on the test = Smishing!
  • Vishing — voice phishing; talking someone into doing something over a phone call

File-Based Attacks

  • Embedded macros, malicious PDFs, Office macros, Trojan installers
  • Fileless malware — no file stays on the system; a downloader pulls malware, runs it, then removes itself
  • Images can be malicious via steganography — hidden malware or exfiltrated data inside image files

Removable Media

  • USB drives, SD cards, external HDDs can exfiltrate data or introduce malware
  • Even air-gapped systems can be compromised via removable media (e.g., Stuxnet on Iranian nuclear facility)
For exam: Disable AutoRun, restrict USB devices, user training

🚨 Indicators of Compromise (IoC)

▼
  • Account Lockout — user account got locked out
  • Concurrent Sessions — same account logged in from multiple places
  • Impossible Travel — login from New York then Paris 10 minutes later
  • Resource Consumption — associated with DDoS
  • Inaccessibility — service disruption (DDoS)
  • Out-of-Cycle Logging — logs generated at unexpected times; verify timestamps match
  • Missing Logs — attacker disabled logging to cover tracks
  • Published/Documented — data exposure or leakage
  • Blocked Content — content being blocked unexpectedly

Linux File Permissions (chmod)

Read=4, Write=2, Execute=1
Format: chmod [owner][group][other]

chmod 777 = owner:rwx · group:rwx · other:rwx (all permissions)
chmod 644 = owner:rw · group:r · other:r
chmod 640 = owner:rw · group:r · other: none
ACS = Access Control System · ACL = Access Control List · ACE = Access Control Entry (ACEs make ACLs)

Least Privilege

  • Each entity gets only the minimum permissions needed to perform its function
  • Mitigates damage from disgruntled employees and compromised accounts

🛡️ Hardening, Patching & Endpoint Protection

▼

Hardening

  • Securing a system by reducing its attack surface — removing unnecessary things from default state
  • Disable unnecessary ports and protocols
  • Change default passwords
  • Remove unnecessary software

Patching

  • Systematic notification, identification, deployment, installation, and verification of updates
  • Updates = patches, hotfixes, service packs
  • Must be kept current to address newly discovered vulnerabilities

Encryption at Rest & Transit

TypeExampleNote
FDE (Full Disk Encryption)BitLocker (Windows), FileVault (Mac)Physical security — protects data on the device
SED (Self-Encrypting Drive)Hardware FDE — encrypts data as it enters/leaves the driveNo burden on host OS
VPNTrusted endpoint encryption over untrusted networkData in transit
Email EncryptionPGP, GPG, S/MIMEData in transit

HIDS vs HIPS

SystemActionExam Tip
HIDSDetection ONLY — monitors and alerts, does not stop threats"D" = Detect
HIPSDetection AND Prevention — actively stops threats as they occur"P" = Prevent ("Hips don't lie")

Configuration Enforcement

  • Secure Baseline — defined secure initial state; used for monitoring and compliance
  • Secure Configuration Guides — vendor-specific instructions enforced through policy
  • Configuration Validation — scripts verify configurations match the approved baseline (think: verification)
  • Decommissioning — when removing a device, purge all sensitive data
    • Spinning/magnetic HDD → degauss or shred
    • SSD → cannot be degaussed; securely erase or shred
    • Routers/switches → reset and purge config

Microservices & Serverless

  • Microservices — Application broken into small, independent services; each does one specific function. Easier to scale and update individually.
  • Serverless (FaaS — Function as a Service) — Code runs in short-lived cloud functions; no server management needed. Provider handles all scaling. Example: AWS Lambda, Azure Functions.
Security note: Containers share the OS kernel — a kernel vulnerability can affect all containers on the host. VMs offer stronger isolation.

🧱 Infrastructure as Code (IaC)

▼
  • Automates provisioning and configuration of infrastructure using code (scripts / templates)
  • Examples: Terraform, Ansible, AWS CloudFormation
  • Benefits: Consistent, repeatable deployments; faster; version-controlled and auditable
  • Security risk: A single misconfiguration in IaC code can be deployed at scale instantly — amplifying mistakes across every environment
  • IaC templates should be scanned and reviewed before deployment (using SAST / policy tools)
Exam tip: IaC = consistency + speed + auditable. The key risk = misconfiguration deployed at scale.

🔄 Resilience & Recovery

RAID (Redundant Array of Independent Disks)

LevelMethodRedundancy?Capacity Trade-off
RAID 0Striping — data split across disks for speed❌ None — one disk failure = total data lossFull capacity; no overhead
RAID 1Mirroring — exact copy on a second disk✅ Survives one disk failure50% usable (half wasted)
